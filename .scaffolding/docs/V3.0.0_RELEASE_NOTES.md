## 🏗️ Directory Restructure

**Problem Solved**: `.agents/` at project root conflicts with projects that are themselves agent projects.

**Solution**: Relocate ALL template infrastructure to `.scaffolding/`

### Before (v2.x)

```
.agents/                    # ❌ Conflicts with agent projects
  ├── skills/
  ├── agents/
  ├── commands/
  ├── bundles.yaml
  └── workflows.yaml

docs/                       # ❌ Mixed scaffold + project docs
  ├── adr/
  └── [scaffold docs]
```

### After (v3.0.0)

```
.scaffolding/
  ├── agents/               # ✅ All template infrastructure
  │   ├── skills/           # 14 skills
  │   ├── agents/           # 6 specialized agents
  │   ├── commands/         # 14 commands
  │   ├── bundles.yaml
  │   └── workflows.yaml
  ├── docs/                 # ✅ All scaffold documentation
  │   ├── adr/              # 13 ADRs (0001-0013)
  │   └── [other docs]
  └── openspec/             # ✅ SDD structure
      ├── project.md
      ├── specs/
      └── changes/

.agents/                    # ✅ Empty - for project use
  └── README.md

docs/                       # ✅ Empty - for project use
  ├── README.md
  └── adr/
      └── README.md
```

**Benefits**:
- ✅ Zero conflicts with agent projects
- ✅ Clear separation (template vs project)
- ✅ AI agents understand context better
- ✅ Easier to sync template updates

## 🚀 Upgrade Process

**One-command migration** from any v2.x version:

```bash
./.scaffolding/scripts/upgrade-to-v3.sh
```

**What it does**:
1. ✅ Checks version compatibility (v2.x → v3.0.0)
2. ✅ Backs up `.agents/` to `.agents.backup/`
3. ✅ Moves `.agents/` → `.scaffolding/agents/`
4. ✅ Creates empty project directories (`.agents/`, `docs/`)
5. ✅ Updates `config.toml` with `[sdd]` section
6. ✅ Updates `AGENTS.md` with new paths
7. ✅ Runs verification checks
8. ✅ Creates git commit with upgrade summary

**Safety features**:
- Dry-run mode: `upgrade-to-v3.sh --dry-run`
- Automatic backups
- Rollback instructions
- Comprehensive error handling

## 🔄 SDD Workflow

**OpenSpec integration** provides structured development workflow:

### Workflow Phases

```
1. brainstorming          → Free-form ideation
2. sdd-propose            → Formalize specs (OpenSpec format)
3. sdd-apply              → Implement with mandatory TDD
4. sdd-archive            → Archive after PR merge
```

### Configuration

Opt-in by default:

```toml
[sdd]
enabled = false           # Opt-in (change to true to enable)
auto_tracking = true      # Auto-create change tracking
```

### Directory Structure

```
.scaffolding/openspec/
├── project.md            # Project metadata
├── specs/                # Formal specifications
│   └── {feature}/
│       ├── proposal.md
│       ├── design.md
│       ├── tasks.md
│       └── STATUS.md
└── changes/              # Implementation tracking
    ├── {change-id}/      # Active changes
    │   ├── implementation.md
    │   ├── tests.md
    │   └── decisions.md
    └── archive/          # Completed changes
```

### Mandatory TDD

**CRITICAL**: `sdd-apply` automatically invokes `test-driven-development` skill:

- ✅ Red-Green-Refactor cycle enforced
- ✅ 80%+ code coverage target
- ✅ Test evidence required before completion
- ❌ Cannot skip testing (Iron Law enforcement)

## 🎓 Iron Laws System

**Philosophy**: "Superpowers strictness × ECC breadth"

Each of the 11 v2.0 skills has **exactly 5 Iron Laws** following standardized format:

### Iron Law Template

```markdown
### {N}. NO {VIOLATION_PATTERN}

**❌ BAD**:
```code
{example showing violation}
```

**✅ GOOD**:
```code
{example showing correct approach}
```

**Violation Handling**: {what to do when violated}
**No Excuses**: {list of common rationalizations to reject}
**Enforcement**: {how to verify compliance}
```

### Example: backend-patterns-v2

1. NO CALLBACK HELL
2. NO UNHANDLED PROMISE REJECTIONS
3. NO SYNCHRONOUS I/O IN ASYNC CONTEXTS
4. NO MISSING ERROR CONTEXT
5. NO HARDCODED CONFIGURATION

## 🔍 Key Architecture Decisions

### ADR 0013 Highlights

**Decision Matrix**:

| Question | Decision | Rationale |
|----------|----------|-----------|
| Keep ECC tdd-workflow? | ❌ Delete | Superpowers test-driven-development is superior |
| Keep both verification skills? | ✅ Keep | Different roles (pre-completion vs continuous) |
| Superpowers author? | Jesse Vincent (@obra) | NOT Kao Chen-Long (OpenSpec author) |
| Where to put template infra? | `.scaffolding/` | Avoid project-level conflicts |
| How many Iron Laws per skill? | Exactly 5 | Balance coverage vs cognitive load |
| SDD default? | Opt-in (disabled) | Allow gradual adoption |

**Integration Principles**:

1. **Workflow over Implementation**: Superpowers skills define workflows, ECC skills define implementation patterns
2. **Complementary Not Competing**: test-driven-development (workflow) + unit-testing-v2 (patterns) work together
3. **Strictness Preserves Intent**: Iron Laws prevent "I'll skip this once" rationalization
4. **Documentation as Infrastructure**: Skills ARE the implementation guide

## 🛡️ Breaking Changes

### ⚠️ BREAKING: Directory Structure

**Impact**: `.agents/` moved to `.scaffolding/agents/`

**Migration Required**: YES

**How to upgrade**:

```bash
# Automatic (recommended)
./.scaffolding/scripts/upgrade-to-v3.sh

# Manual
1. Backup: mv .agents .agents.backup
2. Move: mv .agents.backup .scaffolding/agents
3. Update AGENTS.md references
4. Create empty project directories
```

**Affected Files**:
- `AGENTS.md` - All skill references
- `config.toml` - Skills paths
- `.vscode/settings.json` - Skills paths (if customized)

### ⚠️ BREAKING: Skill Names

**Impact**: ECC skills renamed with `-v2` suffix

**Migration Required**: If you manually reference skills

**Examples**:
- `backend-patterns` → `backend-patterns-v2`
- `frontend-patterns` → `frontend-patterns-v2`
- `api-design` → `api-design-v2`

**Not Affected**:
- Auto-loaded skills (via trigger keywords)
- Superpowers skills (names unchanged)

### Non-Breaking Changes

These changes are **additive** and don't require action:

- ✅ SDD skills (opt-in via config)
- ✅ OpenSpec directory structure (created when enabled)
- ✅ New Iron Laws (enhance existing skills)
- ✅ Documentation updates
